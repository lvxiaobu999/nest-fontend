# 上线健硕性补充清单

本文基于当前项目代码现状整理，目标是回答两个问题：

1. 现在项目已经具备了哪些上线基础能力。
2. 为了让项目在生产环境更稳、更可观测、更容易排障，还需要补哪些能力。

## 1. 当前已经具备的基础能力

从代码上看，下面这些基础设施已经接入：

- 环境变量分环境加载与启动校验
  - `src/app.module.ts`
  - `src/config/configuration.ts`
  - `src/config/env.validation.ts`
  - `src/config/env.util.ts`
- 日志体系
  - `nest-winston` + `winston-daily-rotate-file`
  - 已带 `requestId` / `traceId`
  - 已区分开发和生产日志格式
  - `src/common/logger/winston.logger.ts`
- 请求上下文透传
  - `AsyncLocalStorage` 保存 `requestId` / `traceId`
  - `src/common/context/request-context.ts`
  - `src/common/middleware/request-id.middleware.ts`
- 统一响应结构
  - `src/common/interceptors/response.interceptor.ts`
- 统一异常拦截
  - `src/common/filters/all-exceptions.filter.ts`
- 慢请求日志
  - `src/common/interceptors/logging.interceptor.ts`
- Prisma 数据库接入
  - `src/prisma/prisma.service.ts`
  - `prisma/schema.prisma`
- 构建、单测、容器化数据库基础脚本
  - `package.json`
  - `docker-compose.yml`

结论：项目已经有“可运行、可记录日志、可统一返回”的基础形态，但距离“生产环境健硕”还差几个关键环节，尤其是输入校验、健康检查、优雅停机、安全加固、错误语义和测试闭环。

## 2. 上线前建议优先补齐的内容

下面按优先级拆分。

### P0：建议上线前必须补

#### 2.1 全局输入校验

当前 `CreateUserDto` / `UpdateUserDto` 只有 TypeScript 类型，没有运行时校验，`main.ts` 里也没有看到 `ValidationPipe`。

现状风险：

- 前端或外部调用传入空字符串、非法邮箱、超长字段时，接口会直接进入业务层。
- `PATCH /users/:id` 可以传入任意字段，缺少白名单过滤。
- DTO 只在编译期生效，运行时不会自动拦截脏数据。

建议补充：

- 安装并启用 `class-validator`、`class-transformer`
- 在 `main.ts` 增加全局 `ValidationPipe`
- 推荐配置：
  - `whitelist: true`
  - `forbidNonWhitelisted: true`
  - `transform: true`
  - `forbidUnknownValues: true`
- 给 DTO 增加明确规则
  - `email` 使用 `@IsEmail()`
  - `name` 使用 `@IsOptional()`、`@IsString()`、`@MaxLength()`

优先级说明：这是最直接降低脏数据和无效请求进入数据库的手段。

#### 2.2 健康检查与就绪检查

当前根路由 `GET /` 只返回项目说明，不是真正的健康检查。

现状风险：

- 部署平台无法准确判断实例是否存活。
- 数据库断连时，服务可能仍返回 200，但实际上已不可用。
- 无法区分 `liveness` 和 `readiness`。

建议补充：

- 增加 `/health/live`
  - 仅表示进程存活
- 增加 `/health/ready`
  - 检查数据库连通性，例如执行简单查询
- 如果后续接 Redis、MQ、对象存储，也一起纳入 readiness

推荐输出结构：

- 应用名
- 当前环境
- uptime
- 依赖项状态
- 时间戳

优先级说明：这是容器编排、负载均衡、自动摘流和故障恢复的基础。

#### 2.3 优雅停机与进程退出策略

当前 `main.ts` 里监听了 `unhandledRejection` / `uncaughtException`，但只是打印日志，没有明确退出策略，也没有看到 `enableShutdownHooks()`。

现状风险：

- 应用收到 `SIGTERM` 时，可能没有完整释放数据库连接。
- 进程已进入异常状态，但仍继续对外服务。
- 容器滚动发布时，旧实例可能来不及完成收尾。

建议补充：

- 在 `main.ts` 开启 `app.enableShutdownHooks()`
- 监听 `SIGINT` / `SIGTERM`
- 在严重不可恢复异常下，记录日志后主动退出进程
- 明确关闭顺序
  - 停止接收新请求
  - 等待在途请求结束
  - 释放 Prisma 连接
  - 退出进程

优先级说明：这是避免“半死不活实例”最关键的稳定性能力之一。

#### 2.4 生产安全基线

当前没有看到以下能力：

- `helmet`
- CORS 策略
- 限流
- 载荷大小限制

现状风险：

- 响应头缺少基础安全保护。
- 来源控制不严格时，接口容易被任意域调用。
- 登录、注册、验证码、短信类接口未来容易被刷。
- 超大请求体可能拖垮实例。

建议补充：

- 启用 `helmet`
- 明确 `enableCors` 白名单，而不是全放开
- 接入 `@nestjs/throttler`
- 设置 body size limit
- 如果接口会对外公开，考虑 Nginx / API Gateway 层限流

优先级说明：安全基线最好在项目早期就定好，后补成本更高。

#### 2.5 数据库异常语义化

当前异常过滤器只处理通用 `HttpException` / `Error`，没有针对 Prisma 常见错误码做映射。

现状风险：

- 唯一键冲突可能直接返回 500，而不是 409
- 记录不存在时可能返回数据库错误，而不是 404
- 前端无法根据错误语义稳定处理

建议补充：

- 在异常过滤器或服务层显式处理 Prisma 错误
- 重点覆盖：
  - 唯一约束冲突 -> 409
  - 记录不存在 -> 404
  - 外键约束失败 -> 400 或 409
- 生产环境不要把内部错误原文直接返回给客户端

优先级说明：这会直接影响接口可用性和调用方体验。

#### 2.6 日志输出策略

当前生产环境下 `Console` transport 被静默，日志只写本地文件。

现状风险：

- 容器环境通常依赖 stdout/stderr 收集日志。
- 如果实例是无状态容器，本地日志文件可能不会长期保留。
- 多实例部署后，排障会变困难。

建议补充：

- 明确生产日志策略
  - 如果跑在容器里，优先输出到 stdout，再由平台采集
  - 本地文件日志作为补充，而不是唯一出口
- 错误日志保留 stack
- 业务日志增加关键字段
  - route
  - method
  - statusCode
  - duration
  - requestId
  - userId（未来有鉴权时）

优先级说明：生产排障时，没有日志等于没有证据。

#### 2.7 测试闭环与 CI 守门

当前情况：

- `pnpm run test` 可通过
- `pnpm run build` 可通过
- `pnpm run lint` 有 6 个 warning
- `pnpm run test:e2e` 失败，原因是 `test/jest-e2e.json` 未处理 `src/...` 绝对导入

现状风险：

- e2e 脚本存在，但现在无法真正拦截回归。
- 关键错误路径没有测试覆盖。
- 目前测试基本只覆盖了根接口 happy path。

建议补充：

- 修复 e2e 路径映射问题
- 增加以下测试：
  - DTO 校验失败
  - 唯一键冲突
  - 查询不存在资源
  - 异常过滤器输出结构
  - 健康检查接口
  - Prisma 连接异常时的 readiness 行为
- 把以下流程接入 CI
  - `pnpm run lint`
  - `pnpm run test`
  - `pnpm run test:e2e`
  - `pnpm run build`

优先级说明：没有 CI 守门，项目会在多人协作和持续迭代中越来越脆。

### P1：建议第一轮上线后尽快补

#### 2.8 接口超时、重试与慢查询治理

当前只有慢请求日志，没有真正的超时保护。

建议补充：

- 为外部依赖调用设置超时
- 为数据库慢查询建立阈值观察
- 对幂等读操作评估重试策略
- 给关键接口建立延迟告警

#### 2.9 分页、过滤和查询边界

当前 `findAll()` 直接全量查表。

建议补充：

- 默认分页
- 设置最大 `limit`
- 明确排序字段
- 防止大表扫描拖垮接口

#### 2.10 配置项分级与文档化

建议把配置分成：

- 必填
- 可选
- 仅开发环境使用
- 仅生产环境使用

同时增加：

- 默认值说明
- 示例值
- 敏感项说明
- 修改后是否需要重启

#### 2.11 监控与告警

建议补充：

- 请求总量
- 错误率
- P95 / P99 延迟
- 数据库连接异常次数
- 实例重启次数

如果后续接 Prometheus，可直接暴露 metrics 接口。

#### 2.12 数据备份与恢复演练

如果使用 PostgreSQL，建议尽快明确：

- 备份频率
- 保留周期
- 恢复流程
- 演练责任人

上线不是只有“能跑”，还要保证“出事能恢复”。

### P2：按业务阶段逐步补

#### 2.13 鉴权、审计与权限边界

如果后续不是纯内网服务，建议补：

- JWT / Session 鉴权
- 角色权限
- 审计日志
- 关键操作留痕

#### 2.14 幂等设计

适合用于：

- 创建订单

- 重复提交保护

#### 2.15 API 文档与版本管理

建议补：

- Swagger / OpenAPI
- `/v1` 版本前缀
- 兼容性策略

## 3. 结合当前代码的具体观察

### 3.1 已做得不错的地方

- 环境变量启动即校验，避免带病启动
- 已接 `requestId` / `traceId`，排障意识是对的
- 有统一响应和统一异常返回，前后端对接成本更低
- 日志已考虑滚动切分，不是纯 `console.log`
- Prisma 已在模块销毁时断开连接

### 3.2 当前最明显的缺口

- `main.ts` 没有全局 `ValidationPipe`
- `main.ts` 没有看到 `enableShutdownHooks()`
- `GET /` 不是健康检查接口
- DTO 没有运行时校验注解
- Prisma 错误未做语义映射
- 生产日志没有 stdout 策略
- e2e 测试脚本当前无法跑通
- `findAll()` 没有分页
- 还没有安全中间件和限流

## 4. 建议上线前最小补齐集合

如果你希望先做一版“够稳”的上线，而不是一次补全全部能力，建议至少先完成这 8 项：

1. 全局 `ValidationPipe`
2. DTO 校验注解
3. `/health/live` 与 `/health/ready`
4. `enableShutdownHooks()` + `SIGTERM` 优雅停机
5. Prisma 常见异常映射
6. `helmet` + CORS 白名单 + 基础限流
7. 修复 `test:e2e` 并补 3 到 5 条关键回归用例
8. 把 `lint/test/e2e/build` 接到 CI

## 5. 推荐执行顺序

建议按下面顺序补，收益最高：

1. 输入校验
2. 健康检查
3. 优雅停机
4. Prisma 错误映射
5. 安全中间件与限流
6. e2e 与 CI
7. 监控告警
8. 备份恢复

## 6. 本次检查结论

这个项目已经有不错的“工程起手式”，尤其是环境变量、日志、异常和请求链路追踪方向是对的。但如果目标是稳定上线，当前还不能只靠这几项就放心发布。

最核心的判断是：

- 你已经完成了“基础设施接入”
- 还没有完成“生产稳态闭环”

差的不是代码能不能跑，而是：

- 错误能不能被准确识别
- 实例异常时能不能优雅退出
- 外部系统能不能知道服务是否可用
- 脏数据能不能被挡在入口
- 回归能不能被测试和 CI 提前拦住

把本文的 P0 项补齐后，这个项目会更接近可以放心上线的状态。
